Overview

Cancellations are a silent revenue leak in ride-hailing — every one is a rider who might not rebook. This project digs into ride-level booking data to separate cancellations that are avoidable (supply-side) from ones that aren't (rider-side), and maps demand against driver availability by hour and zone.

Architecture

Raw ride logs → PostgreSQL → cancellation & utilization analysis in SQL/Python → Power BI operations dashboard.

  • Ingestion: ride-level booking logs including timestamps, cancellation reason codes, pickup zone, and driver assignment status.
  • Cleaning: reconciling inconsistent cancellation-reason labels and removing test/duplicate bookings in Python.
  • Analysis: cancellation-rate breakdowns by reason, hour, and zone; driver-utilization ratios by time window.
  • Reporting: a Power BI dashboard for ops to spot zones and hours with a supply-demand mismatch.

Exploratory Data Analysis

The first useful finding came from just counting cancellation reason codes — "no driver found" and "rider cancelled" looked similar in raw volume but tell completely different stories about where the fix needs to happen.

import pandas as pd

rides = pd.read_csv("ride_bookings.csv", parse_dates=["booking_time"])
cancel_breakdown = (
    rides[rides.status == "cancelled"]
    .groupby("cancellation_reason")
    .size()
    .sort_values(ascending=False)
)
print(cancel_breakdown)

rides["hour"] = rides.booking_time.dt.hour
print(rides.groupby("hour").size().sort_values(ascending=False).head(5))

SQL Analysis

Zone-level and hour-level cancellation rates are computed in SQL so the "where and when" question can be answered directly against the source tables.

-- Cancellation rate by pickup zone and hour, supply-side reasons only
SELECT
    pickup_zone,
    EXTRACT(HOUR FROM booking_time) AS hour_of_day,
    COUNT(*) FILTER (WHERE status = 'cancelled' AND cancellation_reason = 'no_driver_found') AS supply_cancels,
    COUNT(*) AS total_bookings,
    ROUND(
        COUNT(*) FILTER (WHERE status = 'cancelled' AND cancellation_reason = 'no_driver_found')::numeric
        / NULLIF(COUNT(*), 0) * 100, 1
    ) AS supply_cancel_rate_pct
FROM fact_ride_bookings
GROUP BY pickup_zone, hour_of_day
ORDER BY supply_cancel_rate_pct DESC;

Power BI Dashboard

A zone-by-hour heatmap is the centerpiece — dark cells mark the specific zone/hour combinations where "no driver found" cancellations spike, which is a far more actionable view for ops than a single daily cancellation rate.

Business Recommendations

  • Three zones accounted for the majority of supply-side cancellations, concentrated in a narrow evening window — recommended targeted driver incentives for that window rather than a blanket city-wide bonus.
  • Rider-side cancellations clustered heavily in the first 2 minutes after booking — consistent with price or ETA shock, suggesting clearer upfront ETA display rather than a driver-supply fix.
  • Recommended tracking supply-side and rider-side cancellation rates as two separate KPIs — blending them into one metric was masking two very different problems.

Lessons Learned

Treating "cancellation rate" as one number was the biggest early mistake. Splitting it by reason code doubled the analysis time but completely changed the recommendation — the fix for a supply problem and the fix for a rider-perception problem are not the same lever.
