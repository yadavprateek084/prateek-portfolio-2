Sales & Revenue Intelligence Dashboard

A retail dataset of ~120,000 transactions, turned into a single executive dashboard that answers three questions leadership actually asks: where is revenue coming from, where is it slipping, and what should we do about it this quarter.

Overview

The brief I set for myself: assume a mid-size retailer wants one dashboard reviewed every Monday morning. It has to load fast, avoid vanity metrics, and point at an action — not just a number. I sourced a transaction-level dataset (~120K rows spanning products, regions, and time), cleaned it, modeled it, and built the reporting layer on top.

Architecture

Raw CSV exports → PostgreSQL staging tables → cleaned & modeled fact/dimension tables → Power BI (DAX measures) → published dashboard.

  • Ingestion: raw transaction, product, and region files loaded into PostgreSQL.
  • Modeling: a star schema — one fact table (transactions) joined to product, region, and date dimensions.
  • Transformation: Python (Pandas) for outlier handling, currency normalization, and missing-value treatment before it ever reaches BI.
  • Reporting: Power BI with DAX measures for YoY growth, contribution margin, and rolling averages.

Exploratory Data Analysis

Before building anything visual, I profiled the data in Python to catch what a dashboard would otherwise quietly hide: duplicate transaction IDs, a handful of negative-quantity return rows miscoded as sales, and a long tail of low-volume SKUs skewing average order value.

# Quick profiling pass before trusting any downstream number
import pandas as pd

df = pd.read_csv("transactions.csv", parse_dates=["order_date"])
print(df.duplicated(subset="transaction_id").sum(), "duplicate transaction IDs")
print(df[df.quantity < 0].shape[0], "negative-quantity rows")

monthly_rev = (
    df.assign(month=df.order_date.dt.to_period("M"))
      .groupby("month")["revenue"].sum()
)
print(monthly_rev.tail(6))

SQL Analysis

Once the schema was clean, most of the recurring business questions were answered directly in SQL rather than re-computed in BI — keeping the DAX layer thin and fast.

-- Revenue contribution by region, current quarter vs. prior quarter
SELECT
    r.region_name,
    SUM(CASE WHEN t.order_date >= date_trunc('quarter', CURRENT_DATE)
             THEN t.revenue END) AS current_qtr_revenue,
    SUM(CASE WHEN t.order_date >= date_trunc('quarter', CURRENT_DATE) - interval '3 months'
             AND t.order_date < date_trunc('quarter', CURRENT_DATE)
             THEN t.revenue END) AS prior_qtr_revenue
FROM fact_transactions t
JOIN dim_region r ON r.region_id = t.region_id
GROUP BY r.region_name
ORDER BY current_qtr_revenue DESC;

Power BI Dashboard

The dashboard is built around a single top strip of KPIs (total revenue, YoY growth, margin, average order value), with drill-down pages for region, product category, and time trend. Every visual has a one-line annotation explaining what "good" looks like, so it doesn't require narration in the meeting.

Business Recommendations

  • Two regions accounted for over 55% of revenue but showed flat YoY growth — flagged as a concentration risk worth a account-level review.
  • A cluster of mid-tier SKUs had high margin but low visibility in current promotions — recommended reallocating a slice of promo budget toward them.
  • Order value dipped predictably in a specific week each month — consistent with a payroll cycle — suggesting a timed promotional push.

Lessons Learned

The biggest time sink wasn't the DAX — it was the data cleaning. I underestimated how much a "clean-looking" CSV export can still hide: duplicate keys and miscoded returns would have quietly inflated revenue by roughly 4% if I'd trusted the dashboard's first draft. Now I profile before I visualize, every time.
