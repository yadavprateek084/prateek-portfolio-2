Overview

Stockouts lose sales; overstock ties up cash. This project builds a per-SKU demand forecast from ~100K historical sales records and translates it into a reorder-point dashboard that flags which SKUs need attention this week — not a generic accuracy metric nobody in ops reads.

Architecture

Historical sales → PostgreSQL → time-series feature engineering & forecasting in Python → forecast + inventory tables → Power BI reorder dashboard.

  • Ingestion: ~100K daily/weekly sales records across products and locations.
  • Feature engineering: rolling averages, seasonality flags, and lag features built in Pandas.
  • Forecasting: a seasonal decomposition + moving-average baseline per SKU, benchmarked against a naive forecast.
  • Reporting: Power BI dashboard combining forecast, current stock, and a calculated reorder point.

Exploratory Data Analysis

Before forecasting anything, I checked which SKUs even had enough history to forecast reliably — a large share had fewer than 8 weeks of sales, which ruled out a single global model and pushed me toward a tiered approach.

import pandas as pd

sales = pd.read_csv("sales_history.csv", parse_dates=["week"])
weeks_per_sku = sales.groupby("sku_id")["week"].nunique()
print((weeks_per_sku < 8).mean(), "share of SKUs with under 8 weeks of history")

# Decompose one representative SKU to check seasonality strength
top_sku = sales[sales.sku_id == sales.sku_id.value_counts().idxmax()]
print(top_sku.set_index("week")["units_sold"].resample("W").sum().tail(12))

SQL Analysis

Reorder points and current stock health are computed in SQL so the dashboard stays fast and the logic is auditable outside of Power BI.

-- Flag SKUs projected to stock out within the forecast lead time
SELECT
    f.sku_id,
    f.forecast_units_next_4wk,
    i.current_stock,
    i.lead_time_days,
    (i.current_stock - f.forecast_units_next_4wk) AS projected_buffer
FROM fact_forecast f
JOIN dim_inventory i ON i.sku_id = f.sku_id
WHERE (i.current_stock - f.forecast_units_next_4wk) < 0
ORDER BY projected_buffer ASC;

Power BI Dashboard

The dashboard's first view is a single ranked list: SKUs sorted by projected stockout risk, with forecast vs. actual sparklines beside each one, so a purchasing manager can triage in under two minutes.

Business Recommendations

  • ~12% of SKUs were consistently over-forecast, tying up capital in slow-moving stock — recommended a quarterly review to prune or discount them.
  • A small set of high-velocity SKUs had reorder lead times longer than their forecast horizon — flagged as the highest stockout-risk group needing a safety-stock buffer.
  • Recommended a tiered forecasting approach: full seasonal models for high-volume SKUs, simple moving averages for low-volume ones, rather than one model for all.

Lessons Learned

The instinct is to reach for the fanciest forecasting model available. Here, a simple seasonal moving average beat a more complex model on the majority of low-volume SKUs simply because there wasn't enough history to fit anything more sophisticated without overfitting. Matching model complexity to data volume mattered more than model sophistication.
