Customer Behavior & RFM Segmentation

Overview

Retention is cheaper than acquisition, but only if you know who to retain. This project scores every customer on three axes — how recently they bought, how often, and how much — then groups them into segments a marketing team can actually act on: Champions, Loyal, At Risk, Hibernating, and Lost.

Architecture

Order history → PostgreSQL → RFM scoring in SQL → segment logic and validation in Python → Power BI segment explorer.

  • Ingestion: order-level history (~50K unique customers) loaded and deduplicated in PostgreSQL.
  • Scoring: Recency, Frequency, and Monetary values computed per customer, then bucketed into quintiles.
  • Segmentation: quintile combinations mapped to five named, business-readable segments.
  • Reporting: a Power BI segment explorer with drill-through to individual customer lists per segment.

Exploratory Data Analysis

Before scoring, I checked the shape of order frequency and recency — a heavily right-skewed frequency distribution meant a naive equal-width bucketing would have crammed 80% of customers into one segment. Quintile-based scoring fixed that.

import pandas as pd

orders = pd.read_csv("orders.csv", parse_dates=["order_date"])
snapshot_date = orders.order_date.max() + pd.Timedelta(days=1)

rfm = orders.groupby("customer_id").agg(
    recency=("order_date", lambda x: (snapshot_date - x.max()).days),
    frequency=("order_id", "nunique"),
    monetary=("order_value", "sum"),
)
print(rfm.frequency.describe())  # confirms heavy right skew

SQL Analysis

The core RFM scores are computed directly in SQL using window functions, so the segmentation logic lives close to the data and refreshes cheaply.

WITH rfm_base AS (
    SELECT
        customer_id,
        CURRENT_DATE - MAX(order_date) AS recency_days,
        COUNT(DISTINCT order_id) AS frequency,
        SUM(order_value) AS monetary
    FROM fact_orders
    GROUP BY customer_id
),
scored AS (
    SELECT *,
        NTILE(5) OVER (ORDER BY recency_days DESC) AS r_score,
        NTILE(5) OVER (ORDER BY frequency ASC) AS f_score,
        NTILE(5) OVER (ORDER BY monetary ASC) AS m_score
    FROM rfm_base
)
SELECT customer_id, r_score, f_score, m_score,
    CASE
        WHEN r_score >= 4 AND f_score >= 4 THEN 'Champions'
        WHEN r_score >= 3 AND f_score >= 3 THEN 'Loyal'
        WHEN r_score <= 2 AND f_score >= 3 THEN 'At Risk'
        WHEN r_score <= 2 AND f_score <= 2 THEN 'Hibernating'
        ELSE 'Lost'
    END AS segment
FROM scored;

Power BI Dashboard

The segment explorer leads with a treemap of segment size vs. revenue contribution, so it's immediately visible when a small segment (like "At Risk") is punching above its weight in historical spend — the strongest argument for prioritizing it.

Business Recommendations

  • The "At Risk" segment was only 9% of customers but 22% of historical revenue — the single highest-leverage group for a win-back campaign.
  • "Champions" showed strong repeat purchase intervals — recommended a loyalty-tier program rather than discounting, since they weren't price-sensitive.
  • "Hibernating" customers had order histories concentrated in one category — suggested a category-specific re-engagement email rather than a generic one.

Lessons Learned

Quintile-based scoring is only as good as the population it's computed over — I initially scored a full multi-year dataset, which buried recent shifts in customer behavior. Recomputing on a rolling 12-month window gave marketing a segment view that matched what they were actually seeing in current campaigns.
